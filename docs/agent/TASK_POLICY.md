## 默认原则

- 默认按“小任务”处理，除非用户明确说明大任务，或任务实际触发架构、数据库、安全、部署等高风险门禁。
- 少说过程，多给结果；不复述已读文档全文，不粘贴长日志。
- 如果必须扩大范围，先暂停并说明原因，不自行扩展。
- 提交和推送分开授权：可以在用户批准后本地 commit；只有用户明确说“push/推送”时，才允许推送远端。

## 任务分级

### 小任务

示例：

- 移动文件、改 import、改文档、改测试引用；
- 修一个局部 bug；
- 补一个小接口字段或参数校验；
- 不改变公共契约的局部实现。

执行规则：

- 不主动启动专业 Agent；
- 不启动 reviewer，除非准备提交、触发高风险门禁或用户明确要求；
- 不使用 brainstorming；
- 不强制 TDD RED；
- 只跑相关测试；
- 最终汇报控制在 300 字以内。

### 中任务

示例：

- 新增一个后端接口；
- 新增一个 service 或数据处理入口；
- 新增一个向量 collection 或小型导入链路；
- 修改一段核心检索编排，但不改变总体架构。

执行规则：

- 可按需启动 tester 或 reviewer；
- 只有涉及目录边界、跨模块公共接口、核心依赖时才启动 architect；
- 只读取必要共享记忆；
- 优先跑相关测试，提交前再考虑扩大回归。

### 大任务

示例：

- 新顶级模块或明显改变依赖方向；
- 数据库表结构、图数据库约束或生产索引变化；
- 替换核心模型、核心检索算法或基础框架；
- 前后端完整功能闭环；
- 部署、权限、密钥和不可逆操作。

执行规则：

- 必须先给方案；
- 按 AGENTS.md 的 architect、database、reviewer、acceptance 门禁执行；
- 明确批准范围、禁止范围、文件所有权和验收标准；
- 不在同一轮混入无关优化。

## 文档读取策略

- 新任务必须读取 `AGENTS.md` 和本文件。
- 只按任务需要读取 `PROJECT_STATE.md`、`TASK_BOARD.md`、`ARCHITECTURE.md`、`DATABASE.md`、`ACCEPTANCE.md`。
- 读取后只用一句话确认，不复述文档内容。
- 给专业 Agent 的上下文只包含本任务必要片段，不转发完整历史对话。

## 测试策略

- 默认测试解释器固定为项目虚拟环境：`D:\pj\vlrag\shopping-qna\.venv\Scripts\python.exe`。
- 除非用户明确要求，不使用 Anaconda base、系统 Python 或其他全局 Python 作为验收环境。
- 如果 `.venv` 缺依赖，先报告并在当前任务范围内补齐必要开发依赖；不要切换到其他解释器规避问题。
- 小任务：相关测试 + 必要静态检查。
- 中任务：相关测试 + 关键链路 smoke；提交前按风险决定是否扩大回归。
- 大任务：相关测试、集成测试、reviewer 审查和必要验收。
- 已知无关失败只记录，不越界修复。

## 汇报格式

最终汇报默认包含：

- 任务大小判断；
- 修改文件；
- 验证结果；
- 是否阻塞；
- 是否建议提交；只有用户明确要求时才建议或执行 push。

除非用户要求，不输出长日志、完整 diff、完整测试列表或重复背景。
